# Decisions for Approval — final v3

В v3 большинство спорных решений уже переведено в безопасную модель. Но эти пункты всё равно стоит утвердить как политику проекта.

## 1. `&Вместо` в CFE

**Решение v3:** `&Вместо` — BLOCKER.

Агент обязан:

1. сначала предложить `&Перед`, `&После`, `&ИзменениеИКонтроль`;
2. если без `&Вместо` нельзя — показать риск обновляемости и альтернативы;
3. получить явное подтверждение пользователя;
4. описать rollback.

**Нужно утвердить:** оставить так или вернуть полный запрет без исключений.

## 2. `.dev.env` как источник параметров

**Решение v3:** `.dev.env` — главный локальный источник для путей, ИБ, платформы, профиля устройства, имени расширения, запуска 1С.

`AGENTS.md`, `.cursor/rules` и skills не должны хранить секреты и локальные пароли.

**Нужно утвердить:** переносим ли все локальные параметры из `infobasesettings.md` в `.dev.env`.

## 3. OpenSpec

**Решение v3:** OpenSpec не используется по умолчанию.

Дефолтный процесс:

```text
memory-bank/tasks.md → plan → creative → implement/build → reflect → archive
```

OpenSpec можно включать только явно, когда нужна формальная спецификация.

**Нужно утвердить:** оставить OpenSpec optional-only.

## 4. Язык правил

**Решение v3:** русский язык правил, потому что пользователь, проекты, ТЗ и 1С-документация в рабочем процессе на русском.

**Нужно утвердить:** оставляем русский или переводим core-rules на английский.

## 5. Жесткость проверки

**Решение v3:** перед сдачей кода агент должен выполнить доступные проверки. Если инструменты недоступны, он обязан явно написать: “автоматическая проверка не запускалась”.

**Нужно утвердить:** требовать ли всегда `deploy_and_test` перед финальным ответом или только для задач, где есть изменение файлов.

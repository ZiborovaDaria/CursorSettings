# Типичные ошибки и ограничения

Накоплено по результатам проектирования MCP-сервера для КД 2.0 (2026-05-20)
и работы с реальной ИБ через MCP-toolkit.

## Транспорт: curl, не urllib (Windows)

Python `urllib.request` на примитивном HTTP-сервере toolkit рвёт соединение
при чтении заголовков ответа (`ConnectionResetError 10054`), даже на тривиальном
коде. curl - толерантнее и работает. Надёжный паттерн: Python формирует JSON
(`json.dumps(ensure_ascii=False)` - корректное экранирование кириллицы) во
временный UTF-8 файл, отправка через `curl --data-binary @file`. Готовый helper:
`C:\EDT\KD\tools\kd_exec.py` (`python kd_exec.py code < file.bsl`).

## execute_code = Выполнить(): только операторы

`execute_code` исполняет код через `Выполнить()`. Это значит:

- **НЕЛЬЗЯ объявлять `Функция`/`Процедура`** - ошибка «Ожидается последовательность
  операторов» в (1,1). Только линейный код + циклы. Хелперы разворачивать в цикл
  по массиву структур, не в функцию.
- **`СтрСоединить` недоступна** (режим совместимости 8.2.13 КД 2.1) - ошибка
  «Процедура или функция с указанным именем не определена». Соединять строки
  вручную циклом с переменной-разделителем.
- Возврат результата - через переменную `Результат` (toolkit её сериализует).
  `Возврат` на верхнем уровне недопустим (не функция).

## Неявная транзакция toolkit - откат при любой ошибке

Toolkit оборачивает `Выполнить()` в транзакцию. Если код упал на ЛЮБОЙ строке
(даже после цикла записи, напр. на `СтрСоединить`) - **все `Записать()` в этом
вызове откатываются**. Поэтому частичных записей не остаётся: при ошибке смотри
ноль созданных объектов, исправляй и запускай заново целиком. Делать скрипты
идемпотентными (проверка `НайтиПоНаименованию` перед созданием).

## Имена объектов/свойств в КД хранятся с ПРЕФИКСОМ типа

В `Справочник.Объекты` поле `Наименование` = полное имя типа: `ДокументСсылка.ПоступлениеНаРасчетныйСчет`,
`СправочникСсылка.Контрагенты`, `ПеречислениеСсылка.ВидыОперацийСписаниеДенежныхСредств`,
`ПланСчетовСсылка.Хозрасчетный`. Поле `Имя` = короткое (`ПоступлениеНаРасчетныйСчет`).
`НайтиПоНаименованию` искать по ПОЛНОМУ имени с префиксом, иначе вернёт пусто. То же для типов в ПКО
(`ТипИсточника`/`ТипПриемника`).

## Иерархия Значений: для перечислений короче, чем для документов

- Документ/Справочник: `Значения.Владелец = Свойства`, `Свойства.Владелец = Объекты` (3 уровня до Объекта).
- **Перечисление**: значения подчинены ОБЪЕКТУ-перечислению напрямую - `Значения.Владелец = Объекты`
  (2 уровня). Запрос значений перечисления: `ГДЕ Знач.Владелец = <Объект-перечисление>`.
  Обращение `Знач.Владелец.Владелец.Владелец` для перечисления упадёт «Поле не найдено».

## Виды операций / ПКЗ (трансформация значений перечислений)

Трансформация перечисления-источника в перечисление-приёмник = **ПКО-перечисление + ПКЗ**:
- ПКО: Источник=Объект-перечисление источника, Приемник=Объект-перечисление приёмника.
- ПКЗ (Владелец=ПКО-перечисление): `Источник`/`Приемник` = ссылки на `Значения`,
  `ЗначениеИсточник`/`ЗначениеПриемник` = строковые имена значений.
- ПКС реквизита-перечисления в документе → `ПравилоКонвертации` = ПКО-перечисление (иначе значение не сконвертируется).
- Если перечисления источника и приёмника ТОЖДЕСТВЕННЫ (как в УХ и БП на одной платформе) - ПКЗ поле-в-поле
  по совпадению имён значений (автонастройка через Соответствие). Если РАЗНЫЕ (как УПП vs БП) - маппинг N→1.

## Поиск НСИ по ГУИД: Поиск=Истина, в XML `Поиск="true"`

Для НСИ-справочников (поиск существующего элемента в приёмнике, без создания) - ПКС с реквизитом-ГУИД
(`Доп_НСИ_ГУИД`): Источник/Приемник=Свойство ГУИД, `Поиск=Истина`. В выгруженном XML атрибут английский -
`<Свойство Поиск="true">`. Grep по «Истина» в XML НЕ найдёт - искать `Поиск="true"`.
ПКО НСИ при этом: `НеЗамещать=Истина`, `НеВыгружатьОбъектыСвойствПоСсылкам=Истина`,
`НеСоздаватьОбъектВПриемникеЕслиНеНайденПоПолямПоиска=Истина`.

## Объекты НЕ находятся при загрузке: связка СинхронизироватьПоИдентификатору + ПродолжитьПоиск

Симптом: документ загрузился, но его ссылочные реквизиты (Контрагент, Договор, Организация,
Валюта, СтатьяДДС, банк/счет) = «Объект не найден», ХОТЯ в ПКО справочника заданы поля поиска
по бизнес-ключу (ПКС с `Поиск=Истина` по `Доп_НСИ_ГУИД`, см. секцию «Поиск НСИ по ГУИД» выше).

Причина: два реквизита ПКО управляют ПОРЯДКОМ поиска объекта в приемнике, и поиск по
внутреннему идентификатору ПЕРЕБИВАЕТ поля поиска:

| СинхронизироватьПоИдентификатору | ПродолжитьПоиск...ЕслиПоИдентификаторуНеНашли | Поведение |
|---|---|---|
| Ложь | (неважно) | поиск ТОЛЬКО по полям поиска (ПКС с Поиск=Истина) - бизнес-ключ работает |
| Истина | Ложь | поиск ТОЛЬКО по внутреннему GUID ссылки; не нашел - стоп, поля поиска НЕ используются |
| Истина | Истина | поиск по GUID, не нашел - переход (fallback) на поля поиска |

Внутренний GUID ссылки в источнике и приемнике РАЗНЫЙ, когда НСИ ведется независимо (не
наполняется через этот обмен). Поэтому при `СинхронизироватьПоИдентификатору=Истина` без
`ПродолжитьПоиск...` объект по GUID не находится, до полей поиска дело не доходит - «Объект
не найден», хотя поля поиска заданы корректно.

Фикс (EXCH-0000002, обмен УХ->БП по УДС): на всех ПКО справочников НСИ с поиском по
бизнес-ключу включить `ПродолжитьПоискПоПолямПоискаЕслиПоИдентификаторуНеНашли=Истина`
(или вовсе снять `СинхронизироватьПоИдентификатору`, если синхронизация по внутреннему
GUID не нужна). В выгруженном XML: `<ПродолжитьПоискПоПолямПоискаЕслиПоИдентификаторуНеНашли>true</...>`
рядом с `<СинхронизироватьПоИдентификатору>true</...>`.

ВАЖНО: правка флага ТОЛЬКО в макете/XML слетит при следующей выгрузке из КД. Флаг
выставлять в САМОМ проекте КД - реквизит ПКО через UI («Продолжить поиск по полям поиска,
если по идентификатору объект не найден») или программно:

```bsl
Объект = ПКО.Ссылка.ПолучитьОбъект();
Объект.ПродолжитьПоискПоПолямПоискаЕслиПоИдентификаторуНеНашли = Истина;
Объект.Записать();
```

Тогда флаг попадет в каждую последующую выгрузку правил.

## Архитектура обмена КД 2.0: движения ОПЦИОНАЛЬНЫ; иначе документ ПРОВОДИТСЯ в приёмнике

Движения регистров передаются **опционально и явно**: для каждого ПКО документа можно указать, какие движения
(по каким регистрам) и как передавать - в ПКО они задаются как **табличные части** (ПКС-группы по регистрам
движений, рядом с ПКС реквизитов и табчастей самого документа). Два режима:
- **Движения ВКЛЮЧЕНЫ в ПКО** → приёмник получает готовые движения из обмена (документ можно не перепроводить).
- **Движения НЕ включены** → они не передаются; документ загружается с `Объект.ОбменДанными.Загрузка = Истина`
  (запись без проведения), а ПОСЛЕ загрузки штатно ПРОВОДИТСЯ механизмом обмена → `ОбработкаПроведения`
  приёмника формирует движения сама по своим счетам/виду операции.

Если движения не передаём (как в УПП->БП и УХ->БП по УДС - в ПКО только реквизиты + табчасти документа,
регистров нет): обработчик `<ПослеЗагрузки>` ставит `ОбменДанными.Загрузка=Истина`, восстанавливает ключевые
реквизиты (Контрагент из Договора, СтатьяДДС из табчасти), маркеры; счета НЕ переносим и НЕ заполняем вручную -
приёмник определит их при проведении по виду операции (поэтому важна трансформация вида операции через ПКЗ).
Глобальные алгоритмы конвертации (`АлгоритмПослеЗагрузкиОбъекта/Данных`) обычно пустые - проведение делает
механизм обмена.

## Язык запросов: BSL-функции в запросах НЕ работают

В тексте запроса (`execute_query`, `Запрос.Текст`) НЕ использовать BSL-функции: `ПустаяСтрока(...)`,
`СтрСоединить`, `СтрРазделить`. Для проверки пустой строки в запросе - сравнение `Поле <> ""`.
`Объект[ИмяРеквизита]` (индексатор по строке) для реквизитов объекта НЕ работает - только через точку
`Объект.ИмяРеквизита` (индексатор есть у Структуры/Соответствия, не у СправочникОбъект).

## execute_code

### Ошибка `Значение не является значением объектного типа (Имя)`

Возникает при попытке прочитать `.Имя` у объекта Тип:

```bsl
// Плохо - у Тип нет свойства Имя
Результат = ТипЗнч(Обр).Имя;
```

Использовать `Строка(ТипЗнч(...))`:

```bsl
// Хорошо
Результат = Строка(ТипЗнч(Обр));
// Например: "Обработка объект: Выгрузка правил обмена"
```

Если нужно именно имя метаданных объекта - использовать `.Метаданные().Имя`:

```bsl
Результат = Обр.Метаданные().Имя;
```

### Запрещённые ключевые слова

В стандартной конфигурации MCP-toolkit блокируется верхнеуровневое
использование:

- `Удалить`
- `Записать`
- `УстановитьПривилегированныйРежим`
- `COMОбъект`
- `УдалитьФайлы`

Это значит, что `СоздатьЭлемент().Записать()` или `Объект.Удалить()` упадут
с ошибкой проверки кода. Решения:

1. **Снять защиту в форме toolkit-а** (правильный путь для разработки на
   тестовой ИБ) - см. форму обработки `MCP_Toolkit.epf`. **Подтверждено:**
   после снятия защиты `Записать` и `УстановитьПометкуУдаления` работают
   корректно.
2. Вызвать через **косвенный путь**: например, через экспортные функции
   общих модулей (если есть).
3. Записать через **вложенный вызов**, который проверка не ловит:

   ```bsl
   ВнутренниеФункции(); // где Записать находится внутри функции

   Функция ВнутренниеФункции()
       Возврат Спр.Записать();
   КонецФункции
   ```

### `Удалить()` висит (timeout)

**Симптом:** `Объект.Удалить()` зависает на 30+ секунд, тоолкит возвращает
timeout. После снятия таймаута объект остаётся в БД (удаление не произошло).

**Причина:** конфигурация работает в режиме модальности `Использовать` -
платформа может ждать подтверждения «Удалить связанные объекты?», но UI нет.

**Решение:** использовать `УстановитьПометкуУдаления(Истина)` вместо
`Удалить()`. Запись помечается на удаление, физически остаётся в БД до
запуска стандартной обработки «Удаление помеченных объектов» в UI КД.

```bsl
// Плохо - висит
Объ = Ссылка.ПолучитьОбъект();
Объ.Удалить();

// Хорошо
Объ = Ссылка.ПолучитьОбъект();
Объ.УстановитьПометкуУдаления(Истина);
// Запись пометки и Записать() вызывается внутри УстановитьПометкуУдаления
```

Альтернатива: для разовой расчистки тестовых записей - удалить через UI КД
(меню «Все функции» -> «Стандартные» -> «Удаление помеченных объектов»).

### Многострочный код в JSON

В `curl -d` многострочный BSL передается через `\n`:

```sh
curl ... -d '{"code":"Стр1;\nСтр2;\nРезультат = ...;"}'
```

Парсер toolkit-а разворачивает `\n` в реальные переносы строк - работает
без проблем (подтверждено).

Альтернатива - использовать heredoc-стиль через временный файл:

```sh
curl -sS -X POST "http://localhost:7003/api/execute_code" \
  -H "Content-Type: application/json" \
  --data-binary @body.json
```

### `Запрос.Текст` должен быть на одной строке

Внутри строкового литерала `Запрос.Текст = "ВЫБРАТЬ ..."` многострочное
форматирование через `\n` ломает парсер. Использовать однострочный текст:

```bsl
// Правильно (одна строка)
Запрос.Текст = "ВЫБРАТЬ Ссылка, Наименование ИЗ Справочник.Контрагенты";

// Плохо - может сломаться
Запрос.Текст = "ВЫБРАТЬ Ссылка,\n   Наименование ИЗ Справочник.Контрагенты";
```

### Функция `ДатаВремя` не существует

В BSL `ДатаВремя` - **токен языка запросов**, не функция платформы:

```bsl
// Плохо
Д = ДатаВремя(2026, 1, 1);

// Хорошо
Д = Дата(2026, 1, 1);
Д = Дата("20260101000000");
```

## Поведение КД 2.0

### ПКО.Код НЕ задавать вручную (генерируется платформой)

Модуль объекта `ПравилаКонвертацииОбъектов.ПередЗаписью` (строка ~84) проверяет
`глИмяКорректно(Код)`. GUID (с дефисами, с цифры) - НЕ корректное имя правила,
запись отклоняется: «Не правильно указано имя правила!» + `Отказ`.

- Для **ПКО/ПКС/ПКЗ** - оставлять `Код` пустым: при пустом Коде `СгенерироватьУникальныйКод()`
  заполнит его из имени объекта-приёмника (`глИмяОбъекта(Приемник)`), напр.
  `ПоступлениеНаРасчетныйСчет00001`, и сделает уникальным.
- Для **Конвертации** - наоборот, `Код` = GUID (40 символов с trailing-пробелами) корректен.

Признак: `Записать()` падает с общим «Не удалось записать "Правила конвертации объектов"!»,
а `ПолучитьСообщенияПользователю()` возвращает 0 сообщений (СообщитьОбОшибке пишет
не в общий стек). Причина почти всегда - некорректный Код или пустой Приемник.

### ПКС-табчасти создавать через СоздатьГруппу()

`ЭтоГруппа` - системное поле, присваивать НЕЛЬЗЯ («Поле объекта недоступно для записи»).
Для ПКС табличной части (Вид=ТабличнаяЧасть) использовать
`Справочники.ПравилаКонвертацииСвойств.СоздатьГруппу()`, для обычного реквизита -
`СоздатьЭлемент()`. Дочерние ПКС реквизитов табчасти: `ПКС.Родитель = <ПКС-группа>`.
Модуль ПКС.ПередЗаписью (при `мУстанавливатьИмяИВидИсточникаПриемника=Истина`, по умолчанию)
сам заполняет ИмяИсточника/ТипИсточника/ВидИсточника/Наименование/Порядок из ссылок
Источник/Приемник - задавать вручную нужно только Источник, Приемник, ПравилоКонвертации
(для ссылочных), Родитель. Тип свойства лежит в табчасти `Свойство.Типы[0].Тип`.

### Программное создание ПКС: ОБЯЗАТЕЛЬНО ссылки, а не текстовые имена

Это самая частая ошибка при создании ПКС через `execute_code`. Если задать только текстовые
`ИмяИсточника` и `ИмяПриемника`, КД в `ПередЗаписью` СБРОСИТ их в пустую строку, если у объекта
в справочнике `Свойства` нет записи с таким именем. ПКС сохранится «мусором» (оба имени пустые),
при выгрузке создаст параметр без имени, и загрузчик в приёмнике упадёт с ошибкой
«**Поле объекта не обнаружено**» в типовом КД-загрузчике (`Обработка.КонвертацияОбъектовИнформационныхБаз`,
строка `ИначеЕсли НЕ ЭтоПараметрДляОбъекта`).

ПРАВИЛЬНО: задавать ССЫЛКИ `Источник` и/или `Приемник` (`Справочник.Свойства`), искать через
`НайтиПоНаименованию` подчинённого объекту. Сигнатура: `(Наименование, ТочноеСоответствие, Родитель, Владелец)` - Владелец это **4-й параметр** (3-й Родитель пропускать).

```bsl
// Получить ссылку на ПКО документа (источник/приёмник = объект в справочнике Объекты)
ОбъектИсточник = ПКО.Источник;  // CatalogRef.Объекты приёмной/исходной конфигурации
ОбъектПриемник = ПКО.Приемник;

// Найти ссылку Свойства "Доп_НомерАСУФХД" подчинённого приёмному объекту
СвПриемник = Справочники.Свойства.НайтиПоНаименованию("Доп_НомерАСУФХД", Истина, , ОбъектПриемник);
Если СвПриемник.Пустая() Тогда
    // Свойство ещё не загружено в КД - выполнить ЗагрузкаСтруктурыМетаданных для нужной
    // конфигурации перед созданием ПКС
    ВызватьИсключение "Свойство не найдено в КД-метаданных";
КонецЕсли;

ПКС = Справочники.ПравилаКонвертацииСвойств.СоздатьЭлемент();
ПКС.Владелец = ПКО;
ПКС.Приемник = СвПриемник;  // ИмяПриемника подтянется автоматически
// ПКС.Источник = ... аналогично через НайтиПоНаименованию у ОбъектИсточник, если нужно поле-в-поле
ПКС.АлгоритмПередВыгрузкойСвойства = "Значение = Источник.Номер;";  // если нужен алгоритм
ПКС.Порядок = 1000;
ПКС.Записать();  // имена заполнятся из ссылок
```

После Записать проверять `ПКС.ИмяПриемника` - если пуст, значит свойство не найдено и ПКС записалось «мусором».

### Передача значения перечисления приёмника без ПКО

Если приёмник реквизита - ссылка на перечисление другой ИБ (у источника такого перечисления нет
или соответствие нужно вычислить алгоритмом), **НЕ нужен ПКО плана перечисления и ПКЗ**. Достаточно
в `АлгоритмПередВыгрузкойСвойства` ПКС положить в переменную `Значение` **строку - имя значения
приёмного перечисления**. КД 2.0 на загрузке найдёт значение с таким именем в указанном
перечислении (по типу реквизита-приёмника) и присвоит.

```bsl
// На выгрузке (Источник - объект исходной ИБ, БП-перечисления недоступны)
ИмяВидаОпер = "";
Для Каждого МетаЗн Из Метаданные.Перечисления.ВидыОперацийПоступлениеДенежныхСредств.ЗначенияПеречисления Цикл
    Если Перечисления.ВидыОперацийПоступлениеДенежныхСредств[МетаЗн.Имя] = Источник.ВидОперации Тогда
        ИмяВидаОпер = МетаЗн.Имя;  // например, "ОплатаПокупателя"
        Прервать;
    КонецЕсли;
КонецЦикла;
Если ЗначениеЗаполнено(Источник.ВидОперацииУХ) И Источник.ВидОперацииУХ.Доп_ДляОплатЗаОП Тогда
    Значение = ИмяВидаОпер + "ЗаОбособленноеПодразделение";  // например "ОплатаПокупателяЗаОбособленноеПодразделение"
Иначе
    Значение = ИмяВидаОпер;
КонецЕсли;
// Значение - просто строка, КД сама подставит enum-значение в приёмнике по имени
```

Это работает, потому что КД 2.0 при загрузке для ссылочного приёмника-перечисления ищет значение
по имени в самом перечислении приёмника. ПравилоКонвертации в ПКС в этом случае оставлять пустым.

Когда нужны ПКО перечисления и ПКЗ: если имена значений в источнике и приёмнике принципиально
не совпадают (нужен явный N→M маппинг через таблицу), либо нужна сложная логика конвертации. Для
простых случаев (имена совпадают или вычислимы) - этот вариант со строкой проще.

### Удаление ПКС: ограничения и обходные пути

- `Удалить()` ВИСИТ (модальность подтверждения связанных объектов). Не использовать.
- `УстановитьПометкуУдаления(Истина)` работает, но **помеченные на удаление ПКС всё равно выгружаются**
  в XML правил.
- `Отключить = Истина` (есть реквизит у ПКС) - ПКС помечен как отключённый, **но при выгрузке тоже
  попадает в XML** (проверено практически). НЕ помогает скрыть ПКС из правил.
- Комбинация `УстановитьПометкуУдаления + Отключить` подряд даёт **«Операция не может быть выполнена,
  так как данные объекта модифицированы»** - после изменения реквизита нужен `Записать()`, затем
  отдельный вызов `УстановитьПометкуУдаления`.
- ПРАКТИЧЕСКИ: чтобы лишняя ПКС не выгружалась и не ломала загрузку - **очистить алгоритмы**
  (`АлгоритмПередВыгрузкойСвойства = ""`) и/или физически удалить через UI КД
  («Все функции» → «Удаление помеченных объектов»). Очистка алгоритмов убирает мусорный код из XML,
  но сама запись с пустыми именами всё равно может попасть в выгрузку.

### Метод `.Свойство()` есть только у Структуры, НЕТ у ДокументОбъект/СтрокаТЧ

Частая ловушка. Метод `.Свойство(ИмяРеквизита)` (проверка наличия) есть только у коллекций-структур:
**Структура, Соответствие, ДанныеФормыСтруктура**. У платформенных объектов конфигурации
(`ДокументОбъект`, `ДокументТабличнаяЧастьСтрока`, `СправочникОбъект`, `РегистрСведенийНаборЗаписей` и т.п.)
метода `.Свойство()` **НЕТ**.

В обработчике `АлгоритмПослеЗагрузкиОбъекта` `Объект` - это `ДокументОбъект` (приёмник), элементы
табчасти `Объект.РасшифровкаПлатежа[i]` - `ДокументТабличнаяЧастьСтрока`. На обоих `.Свойство("...")`
упадёт с «**Метод объекта не обнаружен (Свойство)**».

```bsl
// ПЛОХО - на обмене упадёт:
Если СтрокаТЧ.Свойство("СпособПогашенияЗадолженности") И СтрокаТЧ.СпособПогашенияЗадолженности.Пустая() Тогда

// ХОРОШО - реквизиты документа известны из метаданных, проверять не надо:
Если СтрокаТЧ.СпособПогашенияЗадолженности.Пустая() Тогда
    СтрокаТЧ.СпособПогашенияЗадолженности = Перечисления.СпособыПогашенияЗадолженности.Автоматически;
КонецЕсли;
```

Если реквизиты различаются у Поступления/Списания (или их типов) - не использовать `.Свойство()`,
а **разделять логику в коде** (если ЭтоПоступление / ЭтоСписание) и заранее проверять метаданные через
mcp-edt `get_metadata_details`. Та же ловушка ломает и типовые методы БП-формы вроде
`СчетаУчетаВДокументахКлиентСервер.НачатьЗаполнениеРеквизита` - они вызывают
`ПараметрыЗаполнения.ДанныеФормы.Объект.Свойство(...)`, и потому работают только с **ДанныеФормыСтруктура**,
а не с **ДокументОбъект** напрямую.

Когда защитная проверка нужна - использовать `Попытка/Исключение` вокруг присваивания:

```bsl
Попытка
    Объект.Доп_НеизвестныйРеквизит = Значение;
Исключение
    // реквизита нет у этого типа документа
КонецПопытки;
```

### Заполнение типовых счетов учёта в обмене: ТОЛЬКО внеформенный метод

В БП 3.0 (типовая) подсистема «Счета учёта» имеет два региона методов в `СчетаУчетаВДокументах`:
- **`ЗаполнениеВФорме`** - `ЗаполнитьОбъектПриИзменении(ПараметрыЗаполнения)`. Внутри ожидает
  `ПараметрыЗаполнения.ДанныеФормы.Объект.Свойство(...)` - метод **Свойство() есть только у
  `ДанныеФормыСтруктура`**, у `ДокументОбъект` его нет. В обмене (где работаем с ДокументОбъектом
  напрямую) этот метод **упадёт с ошибкой «Метод объекта не обнаружен (Свойство)»**.
- **`ЗаполнениеВнеФормы`** - `Заполнить(Объект, Отбор)`, `ЗаполнитьПередЗаписью(Объект, РежимЗаписи)`.
  Принимают `ДокументОбъект` напрямую. Внутри привилегированный режим, заполнение по правилам
  менеджера документа. Это правильный метод для обработчика `АлгоритмПослеЗагрузкиОбъекта` в КД.

Пример вызова в обработчике ПослеЗагрузки ПКО:
```bsl
Отбор = Новый Структура("НеЗаполненныеРеквизиты", Истина);
СчетаУчетаВДокументах.Заполнить(Объект, Отбор);  // заполнит только пустые реквизиты
```

Признак выбора: если у метода в описании написано «**для контекстных вызовов из формы**» -
он рассчитан на ДанныеФормы и в обмене не подходит. Искать парный метод во внеформенном регионе.

### ПВД (Правила Выгрузки Данных) создаются из ПКО

ПВД - точки входа выгрузки данных (без них выгружаемый объект не «стартует»). Создавать для
документов; НСИ-справочники ПВД не нужны (тянутся по ссылкам из документов). Реквизиты ПВД
(`Справочник.ПравилаВыгрузкиДанных`, Владелец=Конвертации): `ОбъектВыборки` (= ПКО.Источник),
`ПравилоКонвертации` (= ссылка на ПКО), `СпособОтбораДанных = Перечисления.СпособыОтбораДанных.СтандартнаяВыборка`.
ПередЗаписью автозаполняет `ТипОбъектаВыборки`, `ИмяПравилаКонвертации`, `Код` (из имени ОбъектВыборки),
`Наименование`, группу по умолчанию. В выгруженном XML ПВД лежат в `<ПравилаВыгрузкиДанных>` тегами
`<Правило>` (тот же тег, что у ПКО - при подсчёте через XML-парсер учитывать оба блока).

### Выгрузка НЕ пропускает помеченные на удаление; смена Владельца ПКО запрещена

`ВыгрузкаКонвертации.ВыполнитьВыгрузкуПравил()` включает в XML и помеченные на удаление ПКО.
Сменить `Владелец` у ПКО на другую конвертацию платформа не даёт («Значение поля Владелец не
заполнено или заполнено неверно»). `Удалить()` зависает (модальность). Поэтому **дубли/лишние
ПКО надо либо удалять через UI КД** («Все функции» → «Удаление помеченных объектов») до выгрузки,
**либо вырезать блок `<Правило>...</Правило>` из готового XML текстово** (Python). XML остаётся
well-formed; дубль в ИБ удалить отдельно через UI.

### Trailing-пробелы в Код

Реквизит `Код` в `Конвертации` (40 символов) и `ПравилаКонвертацииОбъектов`
(50 символов) - строки **фиксированной длины** с trailing-пробелами:

```
Код = "00000000-0000-0000-0000-000000000001    "  // 40 символов
Код = "Контрагенты                                       "  // 50 символов
```

При сравнениях использовать `СокрЛП()`:

```bsl
// Плохо - не найдёт из-за пробелов
ПКО = Справочники.ПравилаКонвертацииОбъектов.НайтиПоКоду("Контрагенты");

// Хорошо - искать по Наименованию
ПКО = Справочники.ПравилаКонвертацииОбъектов.НайтиПоНаименованию("Контрагенты", Истина, Конв);

// Если приходится сравнивать Коды - СокрЛП
Если СокрЛП(ПКС.КодПравилаКонвертации) = СокрЛП(ПКО.Код) Тогда ...
```

### `#Если Клиент Тогда` не блокирует MCP-toolkit

Метод `ЗагрузкаСтруктурыМетаданных.ВыполнитьЗагрузку()` имеет модуль с
директивой `#Если Клиент Тогда` (строка 17 модуля). Тем не менее, через
`execute_code` MCP-toolkit метод вызывается без проблем - toolkit живет в
клиентском процессе 1С (тонкий или толстый клиент), поэтому клиентское
условное компилирование не блокирует.

Подтверждение: вызов с несуществующим файлом возвращает ошибку «файл не
найден» от метода `ОткрытьФайл`, а не «метод не определен».

### Переменная `мФормаПрогрессора`

Обработка `ВыгрузкаКонвертации` использует переменную модуля
`мФормаПрогрессора.Значение = 0` (строка 1900 в `ВыполнитьВыгрузкуПравил`).
При программном создании через `Обработки.ВыгрузкаКонвертации.Создать()`
эта переменная не инициализирована из формы - но падения не происходит
(проверено: выгрузка 543 КБ XML отработала успешно).

Возможные сценарии падения:
- Если изменится поведение `мФормаПрогрессора.Значение` присваивания на
  Неопределено.
- В таком случае предварительно установить заглушку-структуру (см. ниже).

**Выгрузка КОМПЛЕКТА правил** (`ВыгружатьПравилаКорреспондента=Истина` и/или
`ВыгружатьПравилаРегистрации=Истина`) обращается к прогрессору ШИРЕ, чем
простая выгрузка одной конвертации - кроме `.Значение` еще `.КомментарийЗначения`,
`.МаксимальноеЗначение`, `.КомментарийОбработкиДанных`. Заглушку создавать сразу
со всеми 4 полями:

```bsl
Обр.мФормаПрогрессора = Новый Структура(
    "Значение,КомментарийЗначения,МаксимальноеЗначение,КомментарийОбработкиДанных",
    0, "", 0, "");
```

Проверено на выгрузке 3-файлового ZIP-комплекта (ExchangeRules +
CorrespondentExchangeRules + RegistrationRules) для БСП-обмена, EXCH-0000002.

### `Состояние(...)` без UI

Платформенная функция `Состояние("текст")` используется в обработках для
показа прогресса. Без UI она просто игнорируется - падения не вызывает.

## execute_query

### Поля ссылочного типа в результатах

В ответах поля ссылочного типа возвращаются как объект `_objectRef`:

```json
{
  "_objectRef": true,
  "УникальныйИдентификатор": "00000000-...",
  "ТипОбъекта": "СправочникСсылка.Конвертации",
  "Представление": "УПП -> БП КОРП Файлы"
}
```

Использовать это как `params` для следующего запроса:

```sh
curl -sS -X POST "http://localhost:7003/api/execute_query" \
  -H "Content-Type: application/json" \
  -d '{
    "query": "ВЫБРАТЬ Наименование ИЗ Справочник.Объекты ГДЕ Владелец = &К",
    "params": {
      "К": {
        "_objectRef": true,
        "УникальныйИдентификатор": "00000000-0000-0000-0000-000000000002",
        "ТипОбъекта": "СправочникСсылка.Конфигурации"
      }
    }
  }'
```

### `НайтиПоНаименованию` для иерархических справочников

В подчиненных справочниках (`Объекты`, `Свойства`, `Значения`, `ПКО`, `ПКС`,
`ПКЗ`) `НайтиПоНаименованию` требует указания Владельца, иначе вернет
произвольный элемент:

```bsl
// Плохо - может вернуть ПКО из другой Конвертации
ПКО = Справочники.ПравилаКонвертацииОбъектов.НайтиПоНаименованию("Контрагенты");

// Хорошо - явный Владелец
ПКО = Справочники.ПравилаКонвертацииОбъектов.НайтиПоНаименованию(
    "Контрагенты", Истина, Конв);
```

Сигнатура: `НайтиПоНаименованию(Наименование, ТочноеСоответствие, Родитель, Владелец)`.

## Производительность

### Большое количество ПКС в одной ПКО

В реальных конвертациях типа БП КОРП -> УПП могут быть 100+ ПКС внутри
одной ПКО. При создании через цикл - использовать **транзакцию**:

```bsl
НачатьТранзакцию();
Попытка
    Для Каждого Поле Из Поля Цикл
        ПКС = Справочники.ПравилаКонвертацииСвойств.СоздатьЭлемент();
        ПКС.Владелец = ПКО;
        // ... заполнение
        ПКС.Записать();
    КонецЦикла;
    ЗафиксироватьТранзакцию();
Исключение
    ОтменитьТранзакцию();
    ВызватьИсключение;
КонецПопытки;
```

Без транзакции каждая запись - отдельная транзакция, медленно.

### Выгрузка XML

Реальные конвертации с 100+ ПКО и 800+ ПКС выгружаются в XML за единицы
секунд. Файл 543 КБ - выгружен за ~1 сек на тестовой ИБ. Особых оптимизаций
не требуется.

## Безопасность

### Резервная копия перед массовыми изменениями

КД не имеет «отмены последнего действия». Перед массовыми изменениями:

1. Скопировать `1Cv8.1CD` в безопасное место (файловая ИБ - просто `cp -r`).
2. Или выгрузить текущие правила через `ВыполнитьВыгрузкуПравил()` как
   backup.
3. Работать на тестовой копии, не на production.

### Привилегированный режим

Для системных операций (создание элементов справочников, запуск выгрузки)
может потребоваться `УстановитьПривилегированныйРежим(Истина)`. По
умолчанию в toolkit это ключевое слово заблокировано - снять защиту в
форме обработки.

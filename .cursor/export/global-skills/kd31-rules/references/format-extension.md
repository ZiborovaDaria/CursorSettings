# Расширение формата EnterpriseData в КД 3.1 + workflow

Цель: получить артефакты для расширения формата ED на стороне приёмника
(например УНФ): **модуль менеджера обмена** (общий модуль BSL с правилами) и
**XSD-пакет/версию формата расширения** (собственный namespace поверх базового ED).
Состав формата можно только расширять (добавлять свойства/типы), не удалять и не
переименовывать существующее.

## End-to-end workflow

Шаги со звёздочкой (*) штатно делаются через формы обработок КД 3.1; toolkit там
помогает инспектировать состояние до/после и при возможности вызвать методы.

1. **Загрузить структуру конфигурации-приёмника.** Обработка
   `ЗагрузкаСтруктурыКонфигурацииИзФайловXML` создаёт `Релиз` и заполняет
   `Объекты`/`Свойства`. Можно через форму, но есть и **headless-путь** через
   экспортный метод `ВыполнитьЗагрузкуМетаданных` + фоновое задание - подробный
   проверенный рецепт в `methods.md` п.1 (поддерживает формат EDT-исходников
   напрямую, без выгрузки в XML Конфигуратора). Проверка:
   `ВЫБРАТЬ КОЛИЧЕСТВО(*) ИЗ Справочник.Релизы`.

2. **Привязать Релиз к Конвертации.** Поля `Конфигурация` и `КонфигурацияКорреспондent`
   Конвертации = ссылка на загруженный Релиз (для двустороннего УНФ-УНФ - один и тот
   же Релиз). Можно через toolkit (правка реквизитов после загрузки структуры).

3. **Подготовить базовый формат ED и расширение.** Базовая версия формата ED (например
   1.10) и расширение - записи справочника `ВерсииФормата` (расширение - с
   `Расширение=Истина` и своим `ПространствоИмен`, см. bsl-templates). Состав формата
   хранится НЕ в табличных частях, а в цепочке подчинённых справочников:
   `ВерсииФормата` -> `ОбъектыФормата` (Владелец=ВерсииФормата) -> `СвойстваФормата`
   (Владелец=ОбъектыФормата) -> `ЗначенияФормата`; типы - `ТипыФормата`
   (Владелец=ВерсииФормата). Добавляемые объекты/свойства расширения создаются как
   элементы этих справочников под записью ВерсииФормата с `Расширение=Истина`. Привязать
   версии формата к Конвертации через ТЧ `Конвертации.ВерсииФормата` (реквизит `ВерсияФормата`).

4. **Наполнить правила (*).** `НастройкаПравилОбмена` / `НастройкаПравилКонвертацииСвойств`:
   ПКО (ОбъектКонфигурации <-> ОбъектФормата), ПКС (свойства, включая Демо_-поля
   расширения), ПОД (точки выгрузки), ПКПД (предопределённые/перечисления). Простые
   ПКО/ПКС можно создавать и через toolkit как элементы справочников (см. bsl-templates),
   нестандартную логику - в алгоритмах ПКО (`АлгоритмПриОтправкеДанных`,
   `АлгоритмПриПолученииДанных`).

5. **Сгенерировать артефакты.** Обработка `ВыгрузкаМодуля` формирует модуль
   менеджера обмена. Headless-вызов через экспортный `ВыполнитьВыгрузкуМодулей` +
   `ПодготовитьНастройкиВыгрузкиМодулей` - см. `methods.md` п.2. Структура формата
   выгружается как XSD (`ВыгрузкаСтруктурыФормата` / штатная выгрузка пакета).

6. **Перенести в расширение приёмника.** XSD -> `XDTOPackage`, BSL -> `CommonModule`
   в расширении конфигурации (например УНФ).

7. **Зарегистрировать расширение формата на стороне приёмника.** Перехват типовой
   процедуры в `ОбменДаннымиПереопределяемый`:
   ```bsl
   &После("ПриПолученииДоступныхРасширенийФормата")
   Процедура <Преф>_ПриПолученииДоступныхРасширенийФормата(РасширенияФормата)
       РасширенияФормата.Вставить("<URI расширения>", "<версия базового ED>");
   КонецПроцедуры
   ```
   Перехват по имени процедуры, не по строке (устойчиво к обновлениям).

## Обработчики ПКО в генерируемом менеджере

- Отправка (ИБ -> XDTO): процедура вида `ПКО_<Имя>_ПриОтправкеДанных(ДанныеИБ,
  ДанныеXDTO, КомпонентыОбмена, СтекВыгрузки)` - заполнить `ДанныеXDTO` из `ДанныеИБ`,
  включая поля расширения.
- Получение (XDTO -> ИБ): `ПриКонвертацииДанныхXDTO(ДанныеXDTO, ПолученныеДанные,
  КонвертацияСвойств, КомпонентыОбмена)` - заполнить объект ИБ.

Для ссылочного свойства с динамическим выбором правила значение - структура
`Новый Структура("Значение, ИмяПКО", Ссылка, "<ИмяПКО>")`. Табличную часть через
алгоритм - заполнить таблицу значений и `ДанныеXDTO.Вставить("<ИмяТЧ>", Таб)`.

## Пример конкретного проекта

Иллюстрация привязки к реальной задаче (двусторонний обмен 7.7 <-> УНФ). Значения
специфичны для проекта - в переиспользуемом сценарии заменять своими.

- Конвертация: `Демо_РасширениеEnterpriseData` (заготовка в ИБ КД 3.1).
- URI расширения: `http://example.com/edi/DemoExtension/1.0`; версия формата
  «Расширение Демо 1.0» (`Расширение=Истина`).
- Базовый формат: EnterpriseData (целевая версия 1.10).
- Целевые артефакты: `XDTOPackage.Демо_РасширениеФормата_1_0` +
  `CommonModule.Демо_МенеджерОбменаРасширение` в расширении УНФ `Демо_УНФ_3.Демо`.
- Объекты: Catalog.Номенклатура, Catalog.Контрагенты, Document.РасходнаяНакладная,
  ПриходнаяНакладная, ПеремещениеЗапасов, МаршрутныйЛист, КорректировкаРеализации,
  КорректировкаПоступления.
- Релиз УНФ: на 2026-05-29 загружен headless (см. `methods.md` п.1 - проверенный
  рецепт: Объекты=4060, Свойства=69015, Демо-свойств=202, ~31 мин). В чистой КД,
  где Релизов=0, загрузка структуры - первый шаг workflow.
- Контракт формата и план разработки - в документах проекта
  (`Контракт обмена XML 7.7-УНФ.md`, `План разработки правил обмена XSD и менеджеров 7.7-УНФ.md`).

## Эволюция схемы

Новые поля расширения добавлять как необязательные. Изменение XDTO-пакета на стороне
приёмника требует реструктуризации ИБ - планировать как обновление, не горячую правку.
